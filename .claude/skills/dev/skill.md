# 開発ワークフロー

以下のタスクについて、6つのフェーズを順番に実行してください。
フェーズ1の完了後のみ、結果をユーザーに報告し次のフェーズに進んでよいか確認してください。
フェーズ2→3→4→5→6はユーザー確認なしで連続実行してください（ただし計画レビュー・コードレビューで「要修正」の場合は修正後に続行）。
いずれのフェーズでもユーザーがフィードバックや修正を指示した場合は、それを反映してから次に進んでください。

**タスク:** $ARGUMENTS

---

## フェーズ 1/6: タスク理解と既存実装との矛盾確認

/understand $ARGUMENTS を実行してください。

完了後、以下を行ってからユーザーにタスク理解を報告し「フェーズ2（実装計画）に進みますか？」と**必ず確認**してください（デバイスが1台でもこの確認はスキップしない）。

- `adb devices` を実行して接続中のデバイス一覧を取得する
- 各デバイスについて `adb -s <serial> shell getprop ro.product.model` を実行し、デバイス名（モデル名）を取得する
- デバイス数に応じて**デバイス選択の確認のみ**分岐する:
  - **0台:** デバイスを接続するかユーザーに確認する
  - **1台:** デバイス選択の確認は不要。そのデバイスをインストール対象とし、モデル名を報告するだけでよい
  - **2台以上:** シリアル番号ではなく**デバイス名（モデル名）**で一覧を表示し、どのデバイスにインストールするかユーザーに確認する（フェーズ6で使用）

---

## フェーズ 2/6: 実装計画

ユーザーの承認後、/plan $ARGUMENTS を実行してください。
フェーズ1でのフィードバックがあれば計画に反映してください。

計画をユーザーに報告した後、確認を待たずにフェーズ3に進んでください。

---

## フェーズ 3/6: 実装計画レビュー

/plan-review を実行してください（フェーズ2の計画が対象）。

レビュー結果が「要修正」の場合は、問題点を修正した計画を再作成してください。
レビュー結果に問題がなければ、確認を待たずにフェーズ4に進んでください。

---

## フェーズ 4/6: 実装

/implement $ARGUMENTS を実行してください。
フェーズ2-3で確定した計画に従って実装してください。

変更内容をユーザーに報告した後、確認を待たずにフェーズ5に進んでください。

---

## フェーズ 5/6: コードレビュー

/review を実行してください。

レビュー結果が「要修正」の場合は、指摘事項を修正し、再度 /review を実行してください。
すべての指摘が解消されるまで繰り返してください。

コードレビューに問題がなければ、確認を待たずにフェーズ6に進んでください。

---

## フェーズ 6/6: インストール・動作確認・ドキュメント更新・push

### ステップ1: インストール（フェーズ1で確認したデバイスのみ対象）

```bash
JAVA_HOME="/c/Program Files/Android/Android Studio/jbr" ./gradlew installDebug -Pandroid.inject.device.serial=<フェーズ1で選択されたデバイスのシリアル>
```

※ 複数デバイスが指定された場合やシリアル指定が不要な場合は通常の `installDebug` を使用してください。

インストール完了後、ユーザーに動作確認を依頼し、OKをもらうまで待機してください。

### ステップ2: ドキュメント更新 → コミット → push（動作確認OK後）

ユーザーから動作確認OKをもらった後に実行してください。

1. README.md など関連ドキュメントを今回の変更内容に合わせて更新する（更新不要なら省略可）
2. 変更をコミットする
3. `git push` する

### ステップ3: git差分の掃除（push後）

push完了後、`git status` と `git diff` を実行してワークツリーに未コミットの変更が残っていないか確認してください。

- **差分がない場合:** そのまま完了報告に進む
- **差分がある場合:** 残っている変更の内容を分析し、掃除案（コミットに含める / 変更を破棄する / 別コミットにする等）をユーザーに提示して確認を取ってから対処する

### 完了報告

すべて完了後、最終結果をユーザーに報告してください。
