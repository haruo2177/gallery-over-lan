# 開発ワークフロー

会話の直近のユーザー指示をタスクとして、以下の11フェーズを順番に実行してください。

ユーザー確認が必要なゲートは以下の4つのみです。それ以外のフェーズ間は確認なしで連続実行してください（ただし計画レビュー・コードレビュー・ドキュメントレビューで「要修正」の場合は修正後に続行）。

- **フェーズ1完了後:** タスク理解とデバイス選択の確認
- **フェーズ6完了後:** ユーザーによる動作確認OK
- **フェーズ10:** git差分掃除の対処方針確認
- **フェーズ11:** リリースバージョン番号の確認

いずれのフェーズでもユーザーがフィードバックや修正を指示した場合は、それを反映してから次に進んでください。

---

## フェーズ 1/11: タスク理解と既存実装との矛盾確認

/understand を実行してください。

完了後、以下を行ってからユーザーにタスク理解を報告し「フェーズ2（実装計画）に進みますか？」と**必ず確認**してください（デバイスが1台でもこの確認はスキップしない）。

- `adb devices` を実行して接続中のデバイス一覧を取得する
- 各デバイスについて `adb -s <serial> shell getprop ro.product.model` を実行し、デバイス名（モデル名）を取得する
- デバイス数に応じて**デバイス選択の確認のみ**分岐する:
  - **0台:** デバイスを接続するかユーザーに確認する
  - **1台:** デバイス選択の確認は不要。そのデバイスをインストール対象とし、モデル名を報告するだけでよい
  - **2台以上:** シリアル番号ではなく**デバイス名（モデル名）**で一覧を表示し、どのデバイスにインストールするかユーザーに確認する（フェーズ6で使用）

---

## フェーズ 2/11: 実装計画

ユーザーの承認後、/plan を実行してください。
フェーズ1でのフィードバックがあれば計画に反映してください。

計画をユーザーに報告した後、確認を待たずにフェーズ3に進んでください。

---

## フェーズ 3/11: 実装計画レビュー

/plan-review を実行してください（フェーズ2の計画が対象）。

レビュー結果が「要修正」の場合は、問題点を修正した計画を再作成してください。
レビュー結果に問題がなければ、確認を待たずにフェーズ4に進んでください。

---

## フェーズ 4/11: 実装

/implement を実行してください。
フェーズ2-3で確定した計画に従って実装してください。

変更内容をユーザーに報告した後、確認を待たずにフェーズ5に進んでください。

---

## フェーズ 5/11: コードレビュー

/review を実行してください。

レビュー結果が「要修正」の場合は、指摘事項を修正し、再度 /review を実行してください。
すべての指摘が解消されるまで繰り返してください。

問題がなければ、確認を待たずにフェーズ6に進んでください。

---

## フェーズ 6/11: インストールと動作確認

フェーズ1で確認したデバイスにインストールしてください。

```bash
JAVA_HOME="/c/Program Files/Android/Android Studio/jbr" ./gradlew installDebug -Pandroid.inject.device.serial=<フェーズ1で選択されたデバイスのシリアル>
```

インストール完了後、ユーザーに動作確認を依頼し、**OKをもらうまで待機**してください。
動作確認OK後、フェーズ7に進んでください。

---

## フェーズ 7/11: ドキュメント更新

README.md など関連ドキュメントを今回の変更内容に合わせて更新してください（更新不要なら省略可）。

更新内容をユーザーに報告した後、確認を待たずにフェーズ8に進んでください。
ドキュメント更新が不要だった場合はフェーズ8・9のドキュメントレビューはスキップし、フェーズ9のpushに進んでください。

---

## フェーズ 8/11: ドキュメントレビュー

フェーズ7で更新したドキュメントを対象に /review を実行してください（観点はドキュメントの正確性・整合性・分かりやすさ）。

レビュー結果が「要修正」の場合は、指摘事項を修正し、再度レビューしてください。
すべての指摘が解消されるまで繰り返し、問題がなければフェーズ9に進んでください。

---

## フェーズ 9/11: push

ソースコードとドキュメントの変更をまとめてコミットし、`git push` してください。

push完了後、確認を待たずにフェーズ10に進んでください。

---

## フェーズ 10/11: git差分掃除

`git status` と `git diff` を実行してワークツリーに未コミットの変更が残っていないか確認してください。

- **差分がない場合:** そのまま完了報告に進む
- **差分がある場合:** 残っている変更の内容を分析し、掃除案（コミットに含める / 変更を破棄する / 別コミットにする等）をユーザーに提示し、**確認を取ってから**対処する

差分がクリーンになったらフェーズ11に進んでください。

---

## フェーズ 11/11: GitHub Release

1. `app/build.gradle.kts` の `versionName` / `versionCode` と `git tag --sort=-v:refname | head -5` で既存タグを確認する
2. 今回の変更内容（feat / fix / breaking 等）から次のバージョン番号（semver: `vX.Y.Z`）を提案し、**ユーザーに確認**する
3. 承認後、必要なら `app/build.gradle.kts` の `versionName` / `versionCode` を更新してコミット・push する
4. リリース用 APK をビルドし、フェーズ1で選択したデバイスにインストールする（動作確認は不要、配布物と同じバイナリを実機に入れておくのが目的）:
   ```bash
   JAVA_HOME="/c/Program Files/Android/Android Studio/jbr" ./gradlew installRelease -Pandroid.inject.device.serial=<フェーズ1で選択されたデバイスのシリアル>
   ```
5. タグを作成して push する（5・6 とインストールの順序は前後しても構わない）:
   ```bash
   git tag -a vX.Y.Z -m "vX.Y.Z"
   git push origin vX.Y.Z
   ```
6. リリースノート（今回の変更点を箇条書き）を作成し、`gh release create` で APK を添付してリリースする:
   ```bash
   gh release create vX.Y.Z app/build/outputs/apk/release/app-release.apk \
     --title "vX.Y.Z" --notes "<リリースノート>"
   ```

### 完了報告

すべて完了後、リリースURLを含む最終結果をユーザーに報告してください。
